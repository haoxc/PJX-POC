# 管理目录计划与任务命名对齐整改记录

## 1. 整改依据

本轮整改依据项目规范与管理目录现状审查结果。

核心问题有两类：

- `10-管理` 下的稳定计划与稳定任务长期使用日期型文件名，导致“版本号”与“事件时间”混用。
- 管理文档中的 `aliases` 同时混用自然语言别名、历史口径别名和生命周期路径别名，但缺少统一分层规则。

因此，本轮整改的目标是：将稳定管理对象的文件名、frontmatter 和 `aliases` 入口统一到可持续维护的口径中。

## 2. 整改范围

| 范围          | 结果                                      |
| ----------- | --------------------------------------- |
| 规范基线        | 在 `01-规范` 中补充稳定计划、稳定任务和 `aliases` 的统一规则 |
| 管理入口        | 在 `10-管理` 中补充命名规则与适用边界                  |
| 计划目录        | 将稳定计划统一为 `PLAN-V001-{计划名称}.md`          |
| 任务目录        | 将稳定任务统一为 `{项目标识}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md` |
| frontmatter | 计划补充 `version`，任务补充 `task_id`           |
| aliases     | 区分计划与任务：计划使用“稳定标识别名 + 语义短名别名 + 生命周期路径别名”，任务使用“稳定编号别名 + 语义任务书别名” |
| 文内引用        | 同步更新台账、问题、决策索引和任务书中的计划/任务链接             |
|             |                                         |
|             |                                         |

## 3. 整改结果

### 3.1 命名口径

- 稳定计划统一使用 `PLAN-V001-{计划名称}.md`。
- 稳定任务统一使用 `{项目标识}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md`。
- `V001` 明确表示文档版本号，不再与日期混用。
- `TASK-0001` 明确表示稳定任务编号，文件名中的 `yy.MMdd` 用于表达任务确认或生效时间。

### 3.2 aliases 口径

稳定管理文档的 `aliases` 按对象分层设置：

- 计划：稳定标识别名 + 语义短名别名 + 生命周期路径别名
- 任务：稳定编号别名 + 语义任务书别名

示例：

- `PLAN-V001-POC可行计划`
- `POC可行计划`
- `新奇/计划/POC可行计划/V001`
- `TASK-0001`
- `孪生区域边界表达任务书`

### 3.3 一次性记录边界

以下文档不纳入本次稳定命名整改，继续允许日期型命名：

- 会议纪要
- 执行检查清单
- 审计报告
- 其他一次性过程记录

### 3.4 补充调整：任务命名恢复日期制

在本轮整改完成后，结合当前项目的实际执行方式，对任务命名进一步收口如下：

- 计划属于长期维护型管理对象，继续使用版本号命名。
- 任务属于执行确认型管理对象，在稳定编号之外保留日期更符合管理语义。
- 因此，稳定任务的有效命名口径明确为 `{项目标识}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md`。

本次补充调整的原因是：

- 任务文档天然承载“确认时间”“生效时间”“执行批次”的管理信息。
- 如果任务文件名完全去掉日期，后续在任务重开、任务重写或同编号迭代时，时点信息会被削弱。
- 计划与任务不应套用完全相同的时间表达方式；计划看版本，任务看执行时点，更符合当前项目管理习惯。

补充调整后的有效结论为：

- 计划继续使用 `PLAN-V001-{计划名称}.md`
- 任务使用 `{项目标识}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md`
- 计划 `aliases` 保留“稳定标识别名 + 语义短名别名 + 生命周期路径别名”
- 任务 `aliases` 保留“稳定编号别名 + 语义任务书别名”

## 4. 影响对象

| 类型   | 影响                        |
| ---- | ------------------------- |
| 规范文档 | 命名与 `aliases` 规则被提升为显式规范  |
| 计划文档 | 当前稳定计划改为版本号命名             |
| 任务文档 | 当前稳定任务改为稳定编号加日期命名         |
| 管理索引 | 台账与索引中的引用一并更新             |
| 代理规则 | 同步沉淀到 `.agents` 的规则与演化日志中 |

## 5. 审计结论

本轮整改后，管理目录中的稳定计划、稳定任务和其 `aliases` 已形成统一口径。

命名中的“版本”与“发生时间”已被拆开处理，后续可以更稳定地进行版本升级、跨文档引用和生命周期检索。

## 6. 遗留事项

- `.obsidian/workspace.json` 中的历史打开路径属于工作区缓存，不纳入本次治理整改范围。
- 后续如新增稳定风险台账、问题台账或长期检查基线，可继续沿用计划类对象的“稳定标识 + 语义短名 + 生命周期路径”规则；任务类对象继续采用简化 aliases。
