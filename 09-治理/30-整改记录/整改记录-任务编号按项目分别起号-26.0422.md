# 任务编号按项目分别起号整改记录

## 1. 整改依据

在多项目并行进入任务管理后，原有任务命名规则暴露出一个隐含冲突：

- 原规则默认将 `TASK-0001` 理解为稳定任务编号，但没有明确其稳定范围。
- 如果多个项目同时进入任务管理并继续沿用全局连续编号，会削弱“项目标识”的治理意义。
- 如果多个项目都使用 bare `TASK-0001` 作为检索入口，又会在跨项目场景中产生歧义。

因此，本轮整改的目标是：明确任务编号的稳定范围，区分“项目内稳定编号”和“跨项目稳定外部标识”。

## 2. 整改范围

| 范围 | 结果 |
| --- | --- |
| 规范基线 | 在 `01-规范` 中明确任务编号按项目分别起号 |
| 管理入口 | 在 `10-管理` 中明确跨项目引用口径 |
| 任务台账 | 将新项目任务编号调整为按项目从 `TASK-0001` 起号 |
| 历史任务 aliases | 将 bare `TASK-0001` 调整为项目级稳定标识别名 |
| 代理规则 | 将命名规则同步到 `.agents` |

## 3. 整改结果

### 3.1 编号口径

- 任务编号按项目分别起号，不同项目均从 `TASK-0001` 开始。
- `task_id` 表示项目内稳定编号，不要求跨项目全局唯一。
- 稳定任务文件名继续使用 `{项目标识}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md`。

### 3.2 跨项目引用口径

- 跨项目稳定引用、检索和治理审计时，使用 `{项目标识}-TASK-{四位编号}`。
- 任务 `aliases` 以项目级稳定编号别名为主，不再将 bare `TASK-0001` 作为跨项目默认检索口径。

### 3.3 现状调整

- “数据基座”阶段一任务由 `TASK-0006` 至 `TASK-0008` 回收为 `TASK-0001` 至 `TASK-0003`。
- “新奇”项目历史任务的 bare `TASK-0001` 至 `TASK-0005` alias 调整为项目级别名。

## 4. 影响对象

| 类型 | 影响 |
| --- | --- |
| 规范文档 | 任务编号稳定范围被明确为项目内 |
| 管理文档 | 任务台账与任务规则补充了跨项目引用口径 |
| 任务文档 | 新项目任务编号从 `TASK-0001` 起号，历史任务 alias 调整为项目级口径 |
| 代理规则 | 同步沉淀到 `.agents` 命名规则 |

## 5. 审计结论

本轮整改后，任务编号规则已经从“隐含全局编号”收口为“项目内编号 + 跨项目外部标识”。

这样既保留了项目内部任务管理的清晰性，也避免了多项目并行时的检索歧义。

## 6. 遗留事项

- 后续如新增项目任务，应直接从 `TASK-0001` 起号，不再沿用其他项目的连续编号。
- 后续如需要任务汇总视图，应以 `{项目标识}-TASK-{四位编号}` 为主键显示。
