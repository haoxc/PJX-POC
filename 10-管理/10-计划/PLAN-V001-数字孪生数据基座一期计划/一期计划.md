# 数字孪生数据基座一期计划

## 1. 计划目的

搭建数字孪生数据基座的一期目标，是为了验证和实现三维场景自动化构建生成能力，并支撑孪生体全生命周期管理、第三方平台接口治理以及第三方平台接入管理。

本计划覆盖一期从模块规划、模块设计到整体平台开发落地的工作范围。计划的作用，是为后续任务明确一阶段要做什么、不做什么。

## 2. 一期范围与边界

一期纳入范围：

- 编码标识管理
- 孪生体管理
- 可视化资产管理
- 接口治理管理
- 第三方平台接入管理
- 面向场景自动化构建生成的模块关系与闭环约束
- 整体平台开发工作
- 与黑盒第三方平台的接入实现

一期不纳入范围：

- 版本管理与历史回滚
- 性能优化与运行时治理

## 3. 模块划分

| 模块        | 一期定位 | 一期要解决的核心问题                   | 一期边界           |
| --------- | ---- | ---------------------------- | -------------- |
| 编码标识管理    | 基础模块 | 如何稳定识别对象、表达对象关系、为构建生成提供统一入口  | 不进入具体编码实现和数据迁移 |
| 孪生体管理     | 核心模块 | 如何以孪生体作为最小管理单元支撑全生命周期管理      | 不进入具体场景引擎实现    |
| 可视化资产管理   | 依赖模块 | 如何以引用方式支撑场景生成所需模型、材质、标签等资产消费 | 不展开完整资产平台建设    |
| 接口治理管理    | 约束模块 | 如何治理数据基座与第三方平台之间的输入输出边界      | 不拆解第三方平台内部实现   |
| 第三方平台接入管理 | 接入模块 | 如何对多家平台业务接口进行统一接入、统一配置和统一管理  | 不拆解第三方平台内部实现   |
|           |      |                              |                |

## 4. 黑盒平台约束

UE 和托群 GIS 平台在本计划中都按黑盒外部平台处理。

黑盒约束如下：

- UE 只被视为场景构建生成结果的消费端，不拆解其内部引擎、插件、编辑器逻辑。
- 托群 GIS 平台只被视为空间信息提供端，不拆解其内部数据模型和平台机制。
- 数据基座负责面向黑盒平台完成接入与消费实现，但不承担第三方平台内部能力设计。

## 5. 阶段一：编码标识管理

要解决什么问题：

- 自动化构建生成依赖的对象识别入口是什么。
- 编码标识如何稳定指向孪生体及相关对象。
- 哪些对象必须进入统一标识体系，哪些暂不纳入。

一期基础功能范围：

- 规则库管理：提供 GUI 管理 `KKS` 等编码体系和规则，支持解析逻辑、校验正则、语义映射、规则启停与试运行。
- 编码管理：支持编码数据增删改查、批量导入、智能校验、清洗和冲突识别。
- 编码映射：管理标识与孪生体之间的映射关系，支持候选建议、自动关联和人工确认。
- 构建前校验与异常处理：支持试运行、构建前准入校验、阻断项与警告项识别、异常样本处理。
- 消费输出服务：为后续孪生体管理和场景自动化构建生成输出统一对象、映射结果和校验结果。

需要哪些输入：

- 一期目标与范围边界。
- 现有对象样本与历史命名方式。
- 孪生体作为最小管理单元的既有结论。

应形成哪些设计成果：

- 编码标识管理边界说明。
- 对象分类与标识范围说明。
- 编码与对象关系草案。
- 构建生成所需最小识别链路说明。
- 规则库管理功能范围说明。
- 编码管理功能范围说明。
- 编码映射与自动关联功能范围说明。
- 构建前校验与异常处理说明。
- 后续消费输出对象说明。

如何判断完成：

- 能明确回答哪些对象必须被稳定识别。
- 编码标识与孪生体的关系清楚。
- 后续任务可以据此继续拆解对象规则与约束。
- 能明确回答规则库怎么配、编码怎么管、标识如何映射、异常如何处理。
- 能明确回答编码标识管理模块最终输出什么给孪生体管理和场景生成链路。

哪些内容应延后到后续阶段：

- 版本管理与历史回滚。
- 性能优化与运行时治理。

## 6. 阶段二：孪生体管理

要解决什么问题：

- 孪生体如何作为最小管理单元进入数据基座。
- 孪生体全生命周期管理至少要覆盖哪些信息。
- 孪生体与编码标识、资产引用、外部平台消费之间如何建立稳定关系。

需要哪些输入：

- 阶段一的编码标识边界和对象关系。
- 现有孪生体场景构建统一接口概要设计。
- 典型孪生体对象样本与生命周期节点。

应形成哪些设计成果：

- 孪生体管理边界说明。
- 孪生体全生命周期管理范围说明。
- 孪生体与其他模块的关系说明。
- 面向场景生成的最小孪生体信息清单。

如何判断完成：

- 孪生体作为最小管理单元的边界稳定。
- 生命周期管理范围被清楚表达。
- 后续任务可以据此继续拆解对象管理和构建消费关系。

哪些内容应延后到后续阶段：

- 生命周期高级治理能力。
- 版本管理与历史回滚。
- 性能优化与运行时治理。

## 7. 阶段三：可视化资产管理

要解决什么问题：

- 场景自动化构建生成依赖哪些可视化资产。
- 资产在一期中以什么方式进入数据基座管理。
- 资产与孪生体、构建生成之间如何建立引用关系。

需要哪些输入：

- 阶段二形成的孪生体信息范围。
- 现有模型、材质、标签、面板等资产样本。
- 典型场景生成所需的最小表现资源集合。

应形成哪些设计成果：

- 可视化资产管理边界说明。
- 资产分类与引用范围说明。
- 资产与孪生体关系说明。
- 场景生成所需最小资产消费约束。

如何判断完成：

- 已明确哪些资产属于一期必须管理的范围。
- 资产与孪生体的引用关系已经稳定。
- 后续任务可以据此继续拆解资产接入和消费规则。

哪些内容应延后到后续阶段：

- 复杂资产治理能力。
- 版本管理与历史回滚。
- 性能优化与运行时治理。

## 8. 阶段四：接口治理管理

要解决什么问题：

- 数据基座如何与第三方平台建立稳定边界。
- 哪些输入输出属于数据基座负责，哪些属于黑盒平台负责。
- 场景自动化构建生成链路需要哪些最小接口治理约束。

需要哪些输入：

- 前三阶段形成的对象、孪生体和资产边界。
- 外部平台消费和提供的信息样本。
- 第三方平台接入约束和现有认知。

应形成哪些设计成果：

- 接口治理边界说明。
- 黑盒平台输入输出说明。
- 第三方平台接入约束草案。
- 场景生成链路的最小接口治理清单。

如何判断完成：

- 数据基座与外部平台的责任边界清楚。
- UE 和托群 GIS 平台都能以黑盒方式被表述。
- 后续任务可以据此继续拆解接口设计和接入规则。

哪些内容应延后到后续阶段：

- 第三方平台高级适配能力。
- 版本管理与历史回滚。
- 性能优化与运行时治理。

## 9. 阶段五：第三方平台接入管理

要解决什么问题：

- 如何对接多家第三方平台业务接口并统一管理。
- 如何对平台、接口、认证、地址、映射、日志和状态进行统一纳管。
- 如何让新增平台接入成为标准动作，而不是重复性项目工作。

需要哪些输入：

- 前四阶段形成的模块边界和黑盒平台约束。
- 已知第三方平台接口样本。
- 当前接入方式、认证方式和调用方式样本。
- 数据基座内部标准对象与场景生成消费要求。

应形成哪些设计成果：

- 第三方平台接入管理边界说明。
- 平台接入对象模型草案。
- 多平台接口统一管理范围说明。
- 平台接入、映射、日志、状态管理约束。

如何判断完成：

- 已能明确回答一个平台接入时需要纳管哪些对象。
- 已能清楚说明多平台接口如何被统一管理。
- 后续任务可以据此继续拆解平台接入实现与管理能力。

哪些内容应延后到后续阶段：

- 高级编排与复杂路由能力。
- 版本管理与历史回滚。
- 性能优化与运行时治理。

## 10. 阶段六：场景自动化构建生成闭环收口

要解决什么问题：

- 前五个模块是否已经足以支撑场景自动化构建生成。
- 哪些信息构成最小生成闭环。
- 一期结束时哪些能力稳定、哪些能力留到后续。

需要哪些输入：

- 前五阶段全部设计成果。
- 场景生成目标与验证口径。
- 当前待确认问题和风险。

应形成哪些设计成果：

- 场景自动化构建生成闭环说明。
- 模块间依赖关系说明。
- 一期成果清单。
- 后续实施阶段建议顺序。

如何判断完成：

- 已能清楚说明场景自动化构建生成最小闭环由哪些模块支撑。
- 已能清楚说明一阶段任务该围绕哪些模块展开。
- 已能清楚说明哪些内容留到后续实施阶段。

哪些内容应延后到后续阶段：

- 版本管理与历史回滚。
- 性能优化与运行时治理。

## 11. 一期总体交付结构

| 阶段  | 交付物           | 说明               |
| --- | ------------- | ---------------- |
| 阶段一 | 编码标识管理设计成果    | 明确对象识别入口与标识边界    |
| 阶段二 | 孪生体管理设计成果     | 明确孪生体全生命周期管理边界   |
| 阶段三 | 可视化资产管理设计成果   | 明确资产引用范围与消费关系    |
| 阶段四 | 接口治理管理设计成果    | 明确黑盒平台边界与接入约束    |
| 阶段五 | 第三方平台接入管理设计成果 | 明确多平台接口统一纳管边界    |
| 阶段六 | 场景生成闭环收口成果    | 明确模块闭环、阶段结论和后续顺序 |

## 12. 一期完成判断

一期计划层面的完成，以模块边界、阶段成果和整体平台开发是否形成闭环为判断标准。

完成判断应至少满足：

- 五类核心模块边界清楚。
- UE 和托群 GIS 平台已被稳定视为黑盒平台。
- 已能清楚说明一阶段任务应该先做哪些模块、后做哪些模块。
- 已能清楚说明多平台业务接口如何被统一纳管。
- 整体平台开发工作已经纳入一期实施范围。
- 已能清楚说明哪些内容留到后续阶段。

## 13. 应延后到后续阶段的内容

- 版本管理与历史回滚机制
- 性能优化与运行时治理

## 14. 当前结论

当前更合理的一期推进顺序是：

`编码标识管理 -> 孪生体管理 -> 可视化资产管理 -> 接口治理管理 -> 第三方平台接入管理 -> 场景自动化构建生成闭环收口`

也就是说，一期不仅要完成模块边界和阶段成果定义，还要承接整体平台开发工作，并将多平台业务接口统一管理能力纳入一期；后续阶段主要承接版本管理、历史回滚以及性能与运行时治理。
