# 文档规范与检测规则

## 1. 基本原则

所有项目文档必须符合技术文档规范：准确、简洁、可执行、可追踪。

文档用于支持技术架构设计、模块接口设计、方案评审和后续实施，不写泛泛背景、不写宣传话术、不写无效客套。

## 2. 内容规则

- 一份文档只描述一个明确主题。
- 先定义问题边界，再描述技术方案。
- 必须区分事实、假设、待确认问题和设计决策。
- 需求课题初步分析文档只维护初步分析结论，不记录过程讨论、会议纪要和实现细节。
- 涉及模块时，必须说明职责、输入、输出和依赖关系。
- 涉及接口时，必须说明调用方、提供方、数据对象、错误处理和更新机制。
- 模块接口设计文档必须基于当前有效计划、任务和模块边界撰写，不额外发散到未进入范围的能力。
- 涉及 UE 时，必须区分场景呈现、动画演示、交互控制和视频输出。
- 涉及数据接入时，必须说明现有状态、展示字段、更新方式和业务含义。
- 涉及 POC 时，必须说明交付时间、展示目标、客户诉求和验收口径。

## 3. 表达规则

- 语言保持专业、客观、直接。
- 句子尽量短，避免多层嵌套。
- 使用明确技术名词，避免模糊词。
- 避免重复解释同一背景。
- 避免“先进、强大、智能化、赋能、全面提升”等宣传表达。
- 避免“好的、下面是、我将为你”等对话式填充语。

## 4. 结构规则

- 每份文档必须有一级标题。
- 建议使用 YAML frontmatter 标记文档类型、状态、创建日期和标签。
- 长段落优先拆成短段落或表格。
- 模块边界、职责分工、阶段划分优先使用表格。
- 结论必须可复用，避免只服务当前对话。

## 5. 文档命名与版本规则

文档命名必须区分长期维护型文档、阶段版本快照和一次性过程记录。

| 类型 | 命名原则 | 示例 |
| --- | --- | --- |
| 稳定计划 | 使用 `PLAN-V001-计划名称`；版本写入 frontmatter，不在文件名中使用日期 | `PLAN-V001-数字孪生接口治理阶段规划.md` |
| 下一版本计划 | 升级版本号，不使用日期表达版本 | `PLAN-V002-数字孪生接口治理阶段规划.md` |
| 稳定任务书 | 模块型任务带模块识别，文件名中保留确认日期 | `数据基座-编码标识管理-TASK-0001-编码规则库管理功能-26.0422.md` |
| 一次性过程记录 | 文件名可使用日期，表示发生、确认或发布的时间 | `会议纪要-新奇POC演示严谨事项-26.0417.md` |

规则：

- 长期维护型文档包括规划、规约、规范、索引、路线图、接口标准、模块设计基线等。
- 长期维护型规划文件名使用 `PLAN-V001-{主题}.md`，对应 frontmatter 使用 `version: V001`。
- 如果需要保留多个冻结版本，使用 `V001`、`V002`、`V003` 等版本号作为文件名中的版本段。
- 日期用于记录事件发生时间，不用于表达长期文档版本。
- 模块功能接口设计文档作为设计快照例外，文件名使用 `{功能名称}-接口设计-V001-0422.md`；其中 `V001` 表示版本号，`0422` 表示设计快照日期。
- 模块主文档默认使用 `{模块名称}.md`，可同时承担模块 MOC 与模块接口总览职责，不再强制单独创建 `{模块名称}-接口设计-V001-0422.md`。
- 同一模块下的接口设计文档统一放在模块目录根下，通过文件名前缀区分总览文档与功能文档。
- 稳定入口、索引、台账、规约和持续维护型检查清单不应使用日期作为主要版本标识。
- 模块型稳定任务书文件名使用 `{项目标识}-{模块识别}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md`；其中 `TASK-0001` 是模块内稳定任务编号，`yy.MMdd` 表示任务确认或生效日期；任务稳定编号写入 `task_id` 字段。
- 同一项目下，不同模块的模块型任务分别起号，每个模块都从 `TASK-0001` 开始。
- 支撑型任务可继续使用 `{项目标识}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md`，编号按项目内支撑任务序列维护。
- 模块型任务如需跨项目、跨模块引用、检索或治理审计，使用 `{项目标识}/{模块识别}/TASK-{四位编号}` 作为稳定外部标识。
- 任务默认不使用生命周期路径别名；需要按课题生命周期快速检索时，优先通过计划、模块或台账入口检索。
- 推荐生命周期阶段包括 `规划`、`模块`、`承接`、`设计`、`开发`、`验收`、`发布`、`任务`、`检查`、`问题`、`决策`。
- 生命周期别名用于快速查询，不替代文件名、`type`、`tags` 和稳定 ID。

## 5.1 Aliases 对齐规则

稳定管理文档的 `aliases` 应按以下优先级对齐：

- 第一类：稳定标识别名
  计划至少包含 `PLAN-V001-{计划名称}` 或等价稳定标识。
  模块型任务至少包含 `{项目标识}/{模块识别}/TASK-0001` 这类模块路径稳定 ID。
  支撑型任务至少包含 `{项目标识}-TASK-0001` 这类项目级稳定任务 ID。
- 第二类：语义短名别名
  用于人工搜索，例如 `POC可行计划`、`孪生区域边界表达任务书`。
- 第三类：生命周期路径别名
  默认用于计划，不默认用于任务；使用 `{项目标识}/{类别}/{对象}/{版本或编号}` 形式。

推荐示例：

```text
PLAN-V001-POC可行计划
新奇/计划/POC可行计划/V001
数据基座/编码标识管理/TASK-0001
数据基座/编码标识管理/TASK-0001/编码规则库管理功能任务书
```

## 5.2 计划、任务、模块关联规则

同一项目下，模块型计划、模块型任务和模块必须形成可追踪关联。

最低要求：

- 计划文档如果承接模块实施，必须指出本计划覆盖哪些模块。
- 任务文档如果直接推进模块，必须指出本任务属于哪个计划、推进哪个模块。
- 模块文档必须能指出本模块当前由哪些计划承接、由哪些任务推进。

推荐字段：

```yaml
related_plan:
related_module:
related_tasks:
task_category:
```

推荐做法：

- 计划中增加“关联模块与执行单元”表。
- 模块型任务 frontmatter 中增加 `related_plan` 与 `related_module`。
- 支撑型任务 frontmatter 中增加 `related_plan` 与 `task_category`，`related_module` 可为空。
- 模块 MOC 中增加“关联计划”和“关联任务”区块。

任务分类建议：

- `module-task`
  直接推进某个模块的设计、开发、联调、验证
- `delivery-task`
  提交、打包、汇总、导出、提报
- `governance-task`
  审计、整改、评审、规则整理
- `coordination-task`
  会议、协调、确认、对齐

说明：

- 只有 `module-task` 强制关联模块。
- `delivery-task`、`governance-task`、`coordination-task` 不强制关联模块，但应关联计划或管理对象。
- 例如“POC 提交材料”这类任务属于 `delivery-task`，可以不关联模块。

禁止情况：

- 只能从正文描述猜测任务属于哪个模块。
- 模块型任务已经存在，但任务书没有任何模块指向。
- 计划列出模块目标，但无法追到对应任务。

## 5.3 模块接口设计文档规约

模块接口设计文档用于直接支撑前后端开发、联调和后续维护，默认遵循以下规则：

- 模块主文档可兼作模块接口总览，功能接口设计按功能拆分成独立文档。
- 接口统一使用 `resultApi` 响应风格；分页接口统一使用分页响应体。
- 文档必须先定义核心概念，再展开接口清单、关键接口和数据对象。
- 每个功能接口设计文档至少应包含：设计目标、核心概念、接口清单、关键接口设计、字段级数据字典、数据库表示例、状态码、系统序列图、设计结论。
- 字段级数据字典必须包含字段名、类型、必填、说明和映射关系。
- 映射关系必须明确到表字段；运行态字段必须明确标注“不直接落表”或“运行态聚合对象”。
- 业务对象默认补齐统一审计字段；如对象不落表，需明确说明原因。
- 数据库表示例保留 `CREATE TABLE` 级别说明，用于表达后端表设计；接口设计文档默认不写 `INSERT` 示例。
- 接口文档中使用的状态、来源、质量标记、批次状态等业务枚举，必须显式列出枚举字典。
- 涉及异步处理时，必须定义状态机、查询方式和失败回执字段。
- 涉及逻辑删除、判重、唯一约束、禁止删除条件等实现口径时，必须单独写入“实现约束与业务规则”。
- 涉及黑盒平台时，只定义本系统输入输出和边界，不展开第三方平台内部实现。

## 6. 检测清单

每份文档提交前按以下规则检查：

- 是否能用一句话说明文档目的。
- 是否删除了无效客套和重复背景。
- 是否明确区分已完成内容、当前问题和待补充客户需求。
- 是否明确说明 UE 场景呈现、动画演示视频和系统先进性表达的关系。
- 是否存在未定义的术语或缩写。
- 是否存在无法执行、无法验证的空泛描述。
- 是否保留了后续架构设计或接口设计需要的关键信息。
- 长期维护型文档是否使用稳定名称和版本字段，而不是用日期表达版本。
- 如果是接口设计文档，是否已经补齐核心概念、响应规范、字段字典、映射关系、数据库表示例和序列图。

## 7. 任务命名与检查规则

稳定任务书文件名使用：

模块型任务文件名使用：

```text
{项目标识}-{模块识别}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md
```

支撑型任务文件名使用：

```text
{项目标识}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md
```

当前模块型任务示例项目标识使用 `数据基座`。

示例：

```text
数据基座-编码标识管理-TASK-0001-编码规则库管理功能-26.0422.md
```

规则：

- 模块型任务中的 `TASK-0001` 是模块内稳定任务 ID，不随任务主题变化。
- 同一项目下，不同模块的模块型任务分别起号，每个模块都从 `TASK-0001` 开始。
- 模块型任务的稳定检索与引用使用 `{项目标识}/{模块识别}/TASK-{四位编号}`。
- 任务主题必须见名知意，优先使用业务对象 + 动作结果。
- 模块型任务文件名必须显式写出模块识别，避免同项目不同模块任务混淆。
- 任务文件名中的日期用于表达任务确认或生效时间，不替代 `task_id`。
- 如需记录更细的变更时间，写入 frontmatter 的 `created` 或 `updated` 字段。
- 任务台账、稳定入口和持续维护型检查清单不强制日期。
- 不使用仅有编号的任务书文件名，例如 `TASK-0001.md`。
- 不使用仅有主题的任务书文件名，例如 `孪生区域边界表达.md`。
- 不使用日期前置命名，例如 `26.0417-新奇-TASK-0001-孪生区域边界表达-26.0417.md`。

模块型任务默认采用“功能闭环任务”方式：

- 一个功能对应一张任务书。
- 任务内同时写清设计确认与开发实现。
- 只有高风险、高不确定或需先评审后实施的模块，才额外拆分“前置设计任务”。

功能闭环任务书必须包含：

- 任务目标。
- 功能范围。
- 设计要点。
- 开发内容。
- 输入输出。
- 验收标准。
- 边界说明。
- 风险与处理。

任务检查必须确认：

- 任务是否能独立执行。
- 任务是否能直接指导执行责任人开展设计与开发，而不是只停留在抽象描述。
- 验收标准是否可判断通过或不通过。
- 边界是否明确排除非本任务内容。
- 依赖是否已区分客户输入、内部输入和评审节点。
- 任务台账是否链接到任务书并记录状态、优先级、截止时间和来源。
- 任务是否显式说明所属计划。
- 若为模块型任务，是否显式链接关联模块。
- 模块型任务编号是否在所属模块内顺序维护，而不是跨模块累计。

## 8. 项目默认约束

后续项目文档默认遵循本规则。

如文档内容与本规则冲突，优先修改文档表达；如规则不适用于某类文档，应先补充规则，再继续撰写文档。
