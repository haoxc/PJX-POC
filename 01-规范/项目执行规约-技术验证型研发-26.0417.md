# 项目执行规约：技术验证型研发

## 1. 规约目的

本文档用于定义公司在数字孪生、接口治理、POC 和技术验证类项目中的轻量执行方式。

公司当前研发定位是技术验证性开发，项目重点在于快速验证技术可行性、形成可展示成果、沉淀可复用模块。传统重文档流程不适合当前人手配置，但完全不写文档也会导致需求、设计、开发和验收脱节。

因此，本规约采用“轻文档、强闭环”的方式执行：

- 不追求完整流程文档。
- 不为文档而文档。
- 只保留能驱动开发、降低返工、支撑验收和沉淀复用的核心文档。
- 以模块为最小管理单元，让需求、设计、开发、检查形成闭环。

## 2. 公司执行假设

| 假设         | 对执行方式的影响                      |
| ---------- | ----------------------------- |
| 人手优先       | 流程必须简化，不能依赖专职 BA、架构师、测试、文档工程师 |
| 技术验证型开发    | 项目重点是验证可行性、跑通链路、支撑演示和沉淀方案     |
| 文档重视度较低    | 文档必须短、准、能直接指导开发，否则不会被持续维护     |
| 开发人员为全栈工程师 | 一名开发可同时理解需求、设计接口、实现前后端和完成联调   |
| 需要项目可控     | 必须保留最少关键文档，避免只靠口头沟通推进         |
| 需要模块复用     | 每个模块应留下边界、接口、验收和问题记录，便于后续复用   |

## 3. 执行原则

### 3.1 人手优先原则

项目流程以现有人力可执行为前提。

不设置过重的角色分工，不要求每个阶段都形成大型文档。每个模块只要求保留最低必要信息，确保开发人员能独立理解、实现、检查和交接。

### 3.2 文档驱动研发原则

文档不是汇报材料，而是开发入口。

每个模块开发前，必须先明确：

- 做什么。
- 不做什么。
- 输入是什么。
- 输出是什么。
- 接口或数据边界是什么。
- 怎么判断完成。

这些内容不要求长，但必须写清楚。默认情况下，模块型任务应直接写成可执行的“功能闭环任务”，而不是先拆一张抽象设计任务，再拆一张开发任务。

### 3.3 模块闭环原则

项目不按“大需求、大设计、大开发”推进，而按模块形成小闭环。

一个模块必须完成：

```text
需求确认 -> 模块设计 -> 开发实现 -> 自测检查 -> 验收记录 -> 问题归档
```

只要模块闭环成立，项目整体就可以持续向前推进。

### 3.4 核心文档最小化原则

项目只强制维护 5 类核心文档：

| 文档      | 是否必需 | 作用                    |
| ------- | ---- | --------------------- |
| 需求课题分析  | 必需   | 说明为什么做、解决什么问题、边界在哪里   |
| 模块任务书   | 必需   | 将需求拆成功能闭环任务，明确设计、开发、范围和验收 |
| 模块设计说明  | 必需   | 说明模块职责、输入输出、接口、数据和依赖  |
| 执行检查清单  | 必需   | 检查模块是否完成、是否可演示、是否可交付  |
| 问题与决策记录 | 必需   | 记录关键问题、处理方式和重要设计取舍    |

其他文档按需生成，不作为默认强制项。

## 4. 项目流程总览

公司项目建议采用六步轻量流程。

| 步骤 | 名称 | 目标 | 核心输出 |
| --- | --- | --- | --- |
| 1 | 需求定界 | 明确项目目标、场景、边界和验收方向 | 需求课题分析 |
| 2 | 模块拆分 | 将项目拆成可独立推进的模块单元 | 模块任务书 |
| 3 | 模块设计 | 明确每个模块的职责、接口、数据和依赖 | 模块设计说明 |
| 4 | 开发实现 | 按模块完成前端、后端、数据、集成和演示能力 | 可运行模块 |
| 5 | 检查验收 | 按清单检查功能、接口、数据、演示和风险 | 执行检查清单 |
| 6 | 复盘沉淀 | 记录问题、决策、可复用资产和后续计划 | 问题记录、决策记录 |

## 5. 模块化执行单元

### 5.1 模块定义

模块是公司项目执行的最小闭环单元。

一个模块可以是：

- 一个数字孪生展示场景。
- 一个设备数据接入能力。
- 一个接口治理主题。
- 一个三维场景功能。
- 一个模型调用能力。
- 一个驾驶舱指标组件。
- 一个 POC 演示段落。

模块不按代码目录划分，而按“可交付、可验证、可复用”的工作成果划分。

### 5.2 模块必须具备的六个要素

| 要素 | 说明 |
| --- | --- |
| 目标 | 模块要解决什么问题 |
| 范围 | 本模块做什么，不做什么 |
| 输入 | 需要哪些数据、素材、接口、客户信息或内部依赖 |
| 输出 | 交付什么页面、接口、场景、脚本、文档或演示结果 |
| 设计 | 模块职责、数据结构、接口关系、前后端边界 |
| 验收 | 如何判断模块完成，谁确认，按什么标准确认 |

### 5.3 模块闭环标准

一个模块只有同时满足以下条件，才算闭环：

- 有任务书。
- 有最小设计说明。
- 有实现结果或演示结果。
- 有检查清单。
- 已记录遗留问题。
- 已明确是否进入复用、改造或归档。

## 6. 核心文档体系

### 6.1 文档分层

| 层级 | 文档 | 作用 |
| --- | --- | --- |
| 项目层 | 需求课题分析 | 明确项目背景、目标、边界和验收方向 |
| 计划层 | 阶段规划、执行计划 | 明确阶段、交付物、优先级和风险 |
| 模块层 | 模块任务书、模块设计说明 | 驱动具体开发和联调 |
| 检查层 | 执行检查清单 | 判断是否可提交、可演示、可验收 |
| 治理层 | 问题记录、风险记录、决策记录 | 保留关键过程和取舍依据 |

### 6.2 必写文档

| 文档 | 最低内容 | 建议篇幅 |
| --- | --- | --- |
| 需求课题分析 | 目标、场景、边界、待确认问题 | 1-3 页 |
| 模块任务书 | 目标、功能范围、设计要点、开发内容、输入输出、验收、风险 | 1-2 页 |
| 模块设计说明 | 职责、输入输出、接口、数据、依赖、异常 | 1-3 页 |

模块接口设计文档命名与放置要求：

- 模块主文档默认使用 `{模块名称}.md`，并可直接兼作模块接口总览
- 模块功能接口设计文件名使用 `{功能名称}-接口设计-V001-0422.md`
- 同一模块下的接口设计文档统一放在模块目录根下
- 模块接口设计文档默认统一使用 `resultApi`，并至少包含核心概念、关键接口、字段级数据字典、数据库表示例和系统序列图
- 数据库表示例默认保留 `CREATE TABLE` 级别说明，不默认保留 `INSERT` 示例
| 执行检查清单 | 完成项、未完成项、阻断项、验收结果 | 1 页 |
| 问题与决策记录 | 问题、原因、决策、影响、责任人 | 按条记录 |

### 6.3 可选文档

以下文档按项目复杂度选择，不默认强制：

| 文档 | 触发条件 |
| --- | --- |
| 总体架构设计 | 涉及多个系统、多个模块或长期复用平台 |
| 接口规范 | 涉及多系统集成、外部开放、控制接口或复用接口 |
| 数据字典 | 涉及指标口径、设备点位、时序数据或主数据 |
| 演示脚本 | 涉及 POC、客户汇报、视频交付或销售支持 |
| 测试方案 | 涉及高风险控制、生产系统、性能或安全要求 |
| 运维手册 | 涉及上线运行、客户交付或长期维护 |

### 6.4 长期文档版本规则

长期维护型规划应使用版本化计划标识和 frontmatter 版本字段，不应使用日期作为主要版本标识。

适用对象包括：

- 项目规约。
- 技术规划。
- 阶段规划。
- 接口标准。
- 架构设计基线。
- 模块概要设计基线。
- 长期维护型检查清单。

建议方式：

| 场景 | 建议 |
| --- | --- |
| 持续维护当前版本 | 文件名使用 `PLAN-V001-{主题}.md`，frontmatter 使用 `version: V001` |
| 需要形成下一版 | 文件名升级为 `PLAN-V002-{主题}.md`，frontmatter 使用 `version: V002` |
| 需要生命周期检索别名 | alias 使用 `{项目标识}/计划/{主题}/V001` 或 `{课题域}/{对象或模块}/规划/V001` |
| 记录一次性过程 | 文件名使用 `yy.MMdd` 日期 |

### 6.5 管理文档命名与别名规则

`10-管理` 下的稳定计划和稳定任务必须使用统一命名方式：

| 类别 | 文件名规则 | 必要 frontmatter | 推荐 aliases |
| --- | --- | --- | --- |
| 稳定计划 | `PLAN-V001-{计划名称}.md` | `version: V001` | 稳定标识 + 语义短名 + 生命周期路径 |
| 模块型稳定任务 | `{项目标识}-{模块识别}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md` | `task_id: TASK-0001` + `module_name` | 模块路径稳定 ID + 模块路径语义短名 |
| 支撑型稳定任务 | `{项目标识}-TASK-{四位编号}-{任务名称}-{yy.MMdd}.md` | `task_id: TASK-0001` | 项目级稳定 ID + 语义短名 |
| 一次性过程记录 | `{主题}-{yy.MMdd}.md` | `created` | 语义短名或事件别名即可 |

说明：

- `V001` 是计划文档版本号，不是计划主题的一部分。
- 模块型任务中的 `TASK-0001` 是模块内任务稳定编号，不随任务标题变化。
- 同一项目下，不同模块的模块型任务分别起号，每个模块都从 `TASK-0001` 开始。
- 模块型任务跨项目、跨模块引用时，使用 `{项目标识}/{模块识别}/TASK-{四位编号}` 作为稳定外部标识。
- 任务文件名中的 `yy.MMdd` 用于表达任务确认或生效时间。
- 检查清单、审计报告、会议纪要等过程性记录可以保留日期命名。
- 稳定计划与稳定任务的 `aliases` 不再携带日期；模块型任务默认使用模块路径别名，不再使用仅项目级稳定编号别名。

### 6.6 功能闭环任务规则

模块型任务默认采用“功能闭环任务”方式：

- 一个功能对应一张任务书。
- 任务内同时写清设计确认与开发实现。
- 任务应直接回答执行责任人“这一轮到底开发什么功能”。

默认结构：

- 任务目标
- 功能范围
- 设计要点
- 开发内容
- 输入输出
- 验收标准
- 边界说明
- 风险与处理

只有高风险、高不确定、跨团队评审依赖强，或明确需要先做方案评审再决定是否实施的模块，才额外拆分“前置设计任务”。

### 6.7 计划、任务、模块关联规则

同一项目下，模块实施范围内的计划、任务和模块必须形成显式关联链路：

`计划 -> 任务 -> 模块`

最低要求：

| 对象 | 必须关联 | 推荐实现 |
| --- | --- | --- |
| 模块型计划 | 关联模块 | 在计划文档中维护“关联模块与执行单元” |
| 模块型任务 | 关联计划、关联模块 | 在任务 frontmatter 中维护 `related_plan`、`related_module` |
| 支撑型任务 | 关联计划或管理对象 | 在任务 frontmatter 中维护 `related_plan`、`task_category`，`related_module` 可为空 |
| 模块 | 关联计划、关联任务 | 在模块 MOC 中维护 `related_plan`、`related_tasks` |

执行目的：

- 能从计划追到任务和模块
- 能从模块型任务追到来源计划和目标模块
- 能从支撑型任务追到来源计划或服务对象
- 能从模块反查当前由哪些任务推进

任务分类建议：

- `module-task`
- `delivery-task`
- `governance-task`
- `coordination-task`

说明：

- 只有 `module-task` 强制关联模块。
- 交付、审计、会议、汇总、提交类任务不强制关联模块。
- 例如“POC 提交材料”属于交付型任务，不应被强制挂到某个模块上。

如果模块型计划、模块型任务与模块之间只能依赖人工理解推断，视为治理链路不完整。

## 7. 轻量流程执行细则

### 7.1 需求定界

目标是明确项目为什么做、做成什么样、哪些内容不做。

必须输出：

- 项目目标。
- 业务场景。
- 验收方向。
- 当前假设。
- 待确认问题。
- 不纳入范围。

执行要求：

- 不写泛泛背景。
- 不写宣传性目标。
- 不把所有想法都放入第一阶段。
- 必须明确最低可交付版本。

### 7.2 模块拆分

目标是把项目拆成可执行的小单元。

拆分规则：

- 每个模块应能被一个全栈工程师独立推进。
- 每个模块周期不宜过长，优先控制在 1-5 个工作日。
- 每个模块必须有清晰输入和输出。
- 模块之间依赖必须写明。
- 高风险模块优先拆小。
- 模块一旦进入执行，必须能追到对应计划和模块型任务。

模块优先级建议：

| 优先级 | 定义 |
| --- | --- |
| P0 | 不做则项目无法演示、无法验证或无法交付 |
| P1 | 明显增强效果、稳定性或复用价值 |
| P2 | 可延后优化，不影响当前阶段目标 |

### 7.3 模块设计

目标是让开发前先形成最小设计共识。

模块设计必须说明：

- 模块职责。
- 输入数据。
- 输出结果。
- 前端呈现方式。
- 后端服务或接口。
- 数据结构或字段。
- 依赖系统。
- 异常处理。
- 验收方式。

设计说明可以短，但不能缺少接口和数据边界。

### 7.4 开发实现

开发以模块任务书和模块设计说明为入口。

执行要求：

- 开发过程中发现需求不清，先补任务书或设计说明，再继续实现。
- 接口字段变化，必须同步更新模块设计说明。
- 涉及演示效果的模块，必须保留截图、视频片段或操作说明。
- 涉及数据接入的模块，必须记录数据来源、字段含义和刷新方式。
- 涉及控制或外部开放接口，必须记录权限和风险。
- 若模块型任务与模块发生新增、拆分或调整，必须同步更新计划、任务、模块三处关联关系。

### 7.5 检查验收

模块完成后必须按检查清单确认。

检查维度：

| 维度 | 检查问题 |
| --- | --- |
| 功能 | 是否实现任务书目标 |
| 范围 | 是否出现范围外开发 |
| 数据 | 字段、口径、来源、刷新方式是否清楚 |
| 接口 | 调用方、提供方、异常处理是否清楚 |
| 演示 | 是否能稳定展示或说明 |
| 风险 | 未完成项、阻断项、待确认项是否记录 |
| 交接 | 其他人是否能根据文档理解模块 |

### 7.6 复盘沉淀

复盘不是写总结报告，而是沉淀后续复用所需的信息。

必须记录：

- 本模块是否可复用。
- 哪些设计决策后续不能随意改。
- 哪些问题需要客户确认。
- 哪些接口、数据、素材或组件可以进入资产库。
- 哪些内容应进入下一阶段。

## 8. 角色与责任

在人员有限的情况下，不强制设置复杂角色，但必须明确责任。

| 责任角色 | 可由谁兼任 | 主要职责 |
| --- | --- | --- |
| 项目负责人 | 技术负责人、交付负责人 | 定目标、控范围、定优先级、确认交付 |
| 模块负责人 | 全栈工程师 | 维护任务书、设计说明、实现结果和检查清单 |
| 技术评审人 | 资深工程师、架构负责人 | 检查模块边界、接口、数据和风险 |
| 需求确认人 | 项目经理、客户接口人、业务负责人 | 确认业务目标、验收口径和待确认问题 |
| 交付确认人 | 项目负责人、客户接口人 | 确认是否可提交、可演示、可归档 |

同一人可以兼任多个角色，但责任不能缺失。

## 9. 关键控制点

### 9.1 开发前控制点

开发前必须确认：

- 模块目标是否清楚。
- 输入材料是否足够。
- 输出结果是否明确。
- 不做范围是否明确。
- 验收口径是否能判断通过或不通过。

如果以上内容无法确认，不应直接进入开发。

### 9.2 联调前控制点

联调前必须确认：

- 接口字段已记录。
- 数据来源已记录。
- 调用关系已记录。
- 异常情况已有处理方式。
- 本地或演示环境能跑通基本链路。

### 9.3 提交前控制点

提交前必须确认：

- 检查清单已完成。
- 阻断问题已处理或标注。
- 演示材料可打开、可播放、可说明。
- 待客户确认事项已单独列出。
- 版本、截图、脚本或接口文档已归档。

## 10. 适配不同项目类型

### 10.1 POC 项目

POC 项目以演示和技术验证为目标。

必须保留：

- 需求课题分析。
- POC 执行计划。
- 模块任务书。
- 演示脚本。
- 检查清单。
- 待确认问题清单。

不强制：

- 完整测试报告。
- 运维手册。
- 长篇总体架构设计。

### 10.2 接口治理项目

接口治理项目以规范、分层、接口资产和安全边界为目标。

必须保留：

- 阶段规划。
- 接口资产清单。
- 接口分类标准。
- 接口设计规范。
- 重点接口治理方案。
- 接口评审检查表。

不强制：

- 所有接口一次性补齐。
- 一开始就建设完整治理平台。

### 10.3 数字孪生展示项目

数字孪生展示项目以场景表达、数据映射和演示链路为目标。

必须保留：

- 场景分析。
- 模块任务书。
- 数据字段说明。
- 镜头或交互脚本。
- 执行检查清单。

不强制：

- 所有设备真实接入。
- 所有场景细节一次性精修。

## 11. 执行节奏建议

| 节奏 | 动作 | 输出 |
| --- | --- | --- |
| 每日 | 模块负责人更新任务状态、阻断问题和待确认项 | 任务状态更新 |
| 每 2-3 天 | 检查 P0 模块是否闭环 | 模块检查清单 |
| 每周 | 汇总完成模块、风险、决策和下周重点 | 周度执行记录 |
| 阶段结束 | 归档模块文档、演示材料、问题和决策 | 阶段归档包 |

会议可以少开，但文档入口不能缺失。

## 13. 最低执行标准

当项目时间紧、人手少、客户要求变化快时，至少保留以下内容：

| 最低项 | 要求 |
| --- | --- |
| 一个项目目标 | 明确当前阶段要达成什么 |
| 一个模块清单 | 明确拆成哪些模块，优先级是什么 |
| 每个 P0 模块一份任务书 | 明确目标、范围、输入、输出、验收 |
| 每个 P0 模块一份最小设计说明 | 明确接口、数据、前后端边界 |
| 一份检查清单 | 判断是否可提交或可演示 |
| 一份问题清单 | 记录阻断项和待客户确认项 |

低于该标准，项目将依赖口头记忆推进，后续返工、交接和复用风险会明显增加。

## 14. 规约结论

公司适合采用“模块化、轻文档、强闭环”的项目执行方式。

执行重点不是增加管理流程，而是把少量关键文档放到开发前和验收前。通过模块任务书、模块设计说明和执行检查清单，形成需求、设计、开发、验收的一体化闭环。

该方式符合当前全栈工程师为主、技术验证性开发为主、人手优先的组织状态，也能为后续接口治理、数字孪生模块复用和项目交付沉淀基础资产。
